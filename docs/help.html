limitless-catalog

A single-binary tool that backs up your Limitless Pendant lifelogs into a local, searchable SQLite catalog you own.

Your recordings, your transcripts, your machine. Live API access is fine until it isn't: payloads are large, range queries are awkward, and if the service ever winds down, your history goes with it. This tool pulls everything once, keeps it updated incrementally, and makes it instantly queryable offline: fast full-text search, date and meeting-window lookups, and full export to plain markdown or JSON that any tool can read.

Not affiliated with Limitless. Built by and for pendant owners who want a durable local archive of their own data.

Quick start

1. Get an API key

In the Limitless app: Settings > Developer > create key. Then export it so the tool can find it:

export LIMITLESS_API_KEY=your-key-here

(Or pass -api-key per command; the flag wins over the environment variable.)

2. Build

With Go 1.25+ installed:

go build -o limitless-catalog .

Or cross-compile for any platform (pure Go, no C toolchain needed):

GOOS=linux  GOARCH=amd64 go build -o dist/limitless-catalog-linux-amd64 .
GOOS=darwin GOARCH=arm64 go build -o dist/limitless-catalog-darwin-arm64 .
GOOS=windows GOARCH=amd64 go build -o dist/limitless-catalog-windows-amd64.exe .

3. See what your account has

limitless-catalog span

4. First backfill

Pick a window (all of these are safe to re-run; see Concepts):

limitless-catalog ingest backfill --months 3   # last 3 months
limitless-catalog ingest backfill --all        # everything

5. Keep it current

limitless-catalog ingest incremental

Run that daily; see Scheduling for launchd and cron recipes.

Flag order matters. Global flags (-db, -timezone, -json, -api-key) go before the subcommand, and subcommand flags go before positional arguments:

limitless-catalog -json search --limit 5 "budget review"

Concepts

The local catalog

Everything lands in one SQLite file (default limitless.db, override with the global -db flag). Each lifelog row keeps the title, start and end times in UTC, a local date, duration, speakers, the full transcript as markdown, a category, and the raw API JSON. An FTS5 full-text index over titles and transcripts powers search, and it is updated in the same transaction as every row write, so it can never drift out of sync.

Day bucketing and why --timezone matters

Lifelogs are fetched from the API one local day at a time, and each record is bucketed into a local_date. The same timezone must be used for both the fetch window and the bucketing: if they disagree, recordings near midnight can fall between two days and be silently skipped, and once a day is marked done it will not be refetched. The tool uses your system timezone by default (falling back to UTC when the system zone has no IANA name), but on servers, in containers, and in schedulers the "system" zone is often not what you expect. Pass an explicit IANA zone everywhere that runs unattended:

limitless-catalog -timezone America/Chicago ingest incremental

Dedup: safe re-runs

Records are keyed by the API's lifelog id. On re-ingest, a row whose updatedAt is unchanged is skipped; a changed one is rewritten along with its FTS entry. Re-running any ingest over any window is always safe and cheap: nothing is duplicated, and only genuinely changed records are touched.

Category heuristic

The pendant records background media (TV, podcasts, ads) as well as real conversation. A deliberately conservative heuristic tags obvious cases as media: if the wearer spoke, the log is never media; otherwise it takes two or more distinct broadcast-style markers (phrases like "subscribe", "sponsored by", "this episode") in the transcript. Everything else stays unknown rather than risk mislabeling, which is why most records are unknown.

Resume machinery

Backfills walk their window day by day. Each fully ingested day is recorded with a backfill_done:<date> marker in the ingest_state table, and marked days are skipped on re-runs, so an interrupted backfill picks up where it left off. Today is never marked done (it is still being recorded), and ingest incremental always re-ingests the last stored day too, since it may have been partial when last fetched.

Command reference

General shape:

limitless-catalog [global flags] <command> [command flags] [args]

Global flags

FlagDefaultPurpose
-db stringlimitless.dbPath to the SQLite catalog.
-api-key string$LIMITLESS_API_KEYLimitless API key (only needed by ingest and span).
-timezone stringsystem local, or UTCIANA timezone for day bucketing and API fetch windows.
-jsonoffEmit JSON instead of text (works with every query command).

Global flags always go before the subcommand: limitless-catalog -db /backups/pendant.db recent 5.

ingest

Fetch lifelogs from the Limitless API into the local catalog. Five modes: backfill for a window, incremental for daily top-ups, orphans for records with corrupted timestamps, --date for a single day, and --id for a single record.

ingest backfill walks a date window day by day, marking completed days done so re-runs resume instead of refetching.

FlagDefaultPurpose
--days N90Fetch the last N days.
--months N0Fetch the last N months (overrides --days).
--alloffEverything from the oldest available lifelog to today.
--from-startoffAnchor the window at the oldest available lifelog, going forward (combine with --months to cap it).
--start YYYY-MM-DDExplicit start date (overrides all other window flags).
--end YYYY-MM-DDtodayExplicit end date (requires --start).
limitless-catalog ingest backfill --months 3
limitless-catalog ingest backfill --all
limitless-catalog ingest backfill --from-start --months 6
limitless-catalog ingest backfill --start 2026-01-01 --end 2026-03-31

ingest incremental re-ingests from the last stored day through today. No flags. Requires a non-empty catalog (run a backfill first). This is the command to schedule.

limitless-catalog -timezone America/Chicago ingest incremental

ingest --date fetches exactly one local day, for spot-fixing a gap that stats surfaced.

limitless-catalog ingest --date 2026-06-14

ingest orphans rescues records with corrupted (epoch-zero) startTimes, which no date query can reach. It walks the account's oldest records, ingests every corrupted one with its real start time recovered from the transcript's own line timestamps where possible, and stops at the first record with a plausible date.

limitless-catalog ingest orphans

ingest --id fetches exactly one lifelog by its API id, for spot-fixing a single record.

limitless-catalog ingest --id <lifelog-id>

span

Show the API's oldest and newest lifelogs versus local coverage, so you can see what a backfill would fetch before downloading anything. Needs an API key; suggests a backfill command when local coverage is missing or incomplete.

limitless-catalog span
limitless-catalog -json span

search

Full-text search over titles and transcripts (SQLite FTS5 with Porter stemming, ranked by relevance).

FlagDefaultPurpose
--limit N20Maximum results.
limitless-catalog search "budget review"
limitless-catalog -json search --limit 5 "quarterly forecast"

date

List all logs for one local date. No flags; one positional argument.

limitless-catalog date 2026-07-06

range

List logs for an inclusive date range. No flags; two positional arguments.

limitless-catalog range 2026-07-01 2026-07-06

recent

List the newest logs. One optional positional count (default 10).

limitless-catalog recent
limitless-catalog -json recent 25

meeting

Find logs overlapping a time window, with slack on each side, so a "Q3 planning" meeting scheduled 13:00 to 14:00 still matches recordings that started a few minutes early or ran long. The positional argument is the window start.

FlagDefaultPurpose
--end "YYYY-MM-DD HH:MM"start + 1hWindow end.
--buffer N10Minutes of slack on each side.
limitless-catalog meeting "2026-07-06 13:00"
limitless-catalog meeting --end "2026-07-06 14:30" --buffer 15 "2026-07-06 13:00"

get

Show one log by id (ids appear in every query's output).

FlagDefaultPurpose
--fulloffAlso print the full transcript.
limitless-catalog get AbC123xYz
limitless-catalog -json get --full AbC123xYz

A typical pipeline: -json meeting returns ids, then get --full pulls each transcript. Handy for assistant workflows that debrief a meeting.

export

Write your archive out as plain files any tool can read: one markdown file per lifelog, or a single JSON Lines file (which includes the raw API JSON).

FlagDefaultPurpose
--format md|jsonmdMarkdown files or one lifelogs.jsonl.
--out DIRexportOutput directory.
--start YYYY-MM-DDeverythingOnly export from this date.
--end YYYY-MM-DDOnly export through this date.
--search TERMOnly export conversations matching this phrase.
limitless-catalog export --format md --out export/
limitless-catalog export --format json --start 2026-06-01 --end 2026-06-30
limitless-catalog export --search "budget review" --format md --out export-budget/

stats

Catalog totals, coverage, hours recorded, database size, last ingest time, per-month counts, category breakdown, and any days with zero recordings, so you can spot gaps in a backfill. No flags.

limitless-catalog stats
limitless-catalog -json stats

Scheduling

Run ingest incremental once a day. Always pass an explicit -timezone: launchd and cron jobs frequently run with a different (or missing) system timezone, and a mismatch between the fetch window and day bucketing can silently skip edge-of-day recordings (see Day bucketing).

macOS launchd

Save as ~/Library/LaunchAgents/com.limitless-catalog.plist, then launchctl load ~/Library/LaunchAgents/com.limitless-catalog.plist:

<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE plist PUBLIC "-//Apple//DTD PLIST 1.0//EN"
  "http://www.apple.com/DTDs/PropertyList-1.0.dtd">
<plist version="1.0"><dict>
  <key>Label</key><string>com.limitless-catalog</string>
  <key>ProgramArguments</key><array>
    <string>/usr/local/bin/limitless-catalog</string>
    <string>-timezone</string><string>America/Chicago</string>
    <string>ingest</string><string>incremental</string>
  </array>
  <key>EnvironmentVariables</key><dict>
    <key>LIMITLESS_API_KEY</key><string>your-key-here</string>
  </dict>
  <key>StartCalendarInterval</key><dict>
    <key>Hour</key><integer>6</integer><key>Minute</key><integer>30</integer>
  </dict>
  <key>WorkingDirectory</key><string>/path/to/your/catalog</string>
</dict></plist>

Linux cron

30 6 * * * cd /path/to/catalog && LIMITLESS_API_KEY=your-key ./limitless-catalog -timezone America/Chicago ingest incremental

Direct SQL cookbook

The catalog is plain SQLite with an FTS5 index; query it from anything. Read freely, but write only through the app: the app updates the FTS index in the same transaction as every row write, and out-of-band writes would leave the index stale.

Recent titles:

sqlite3 limitless.db "SELECT local_date, title FROM lifelogs ORDER BY start_utc DESC LIMIT 10"

Full-text search with relevance ranking:

sqlite3 limitless.db "SELECT l.local_date, l.title
FROM lifelogs_fts f JOIN lifelogs l ON l.id = f.id
WHERE lifelogs_fts MATCH 'budget review' ORDER BY rank LIMIT 10"

Talk time per month:

sqlite3 limitless.db "SELECT substr(local_date, 1, 7) AS month,
ROUND(SUM(duration_min) / 60.0, 1) AS hours, COUNT(*) AS logs
FROM lifelogs GROUP BY month ORDER BY month"

Who you talk with most (the speakers column is a JSON array of names):

sqlite3 limitless.db "SELECT je.value AS speaker, COUNT(*) AS logs
FROM lifelogs, json_each(lifelogs.speakers) je
GROUP BY speaker ORDER BY logs DESC LIMIT 10"

Context around a gap day (what surrounds a date stats flagged as empty):

sqlite3 limitless.db "SELECT local_date, COUNT(*) AS logs, ROUND(SUM(duration_min), 0) AS minutes
FROM lifelogs WHERE local_date BETWEEN '2026-06-12' AND '2026-06-16'
GROUP BY local_date ORDER BY local_date"

AI assistants (MCP)

The catalog speaks the Model Context Protocol over stdio, read-only. Any MCP client (Claude Code, Claude Desktop, and others) can search and read your archive; nothing can write to it or trigger network fetches.

Claude Code:

claude mcp add limitless-catalog -- /path/to/limitless-catalog -db /path/to/limitless.db -timezone America/Chicago mcp

Tools exposed: search_transcripts, find_meeting, list_by_date, list_range, recent, get_transcript, catalog_stats.

Read-only by design. The MCP server opens the database with SQLite's read-only mode (mode=ro), so no tool call can modify your archive, and it deliberately exposes no ingest or export tools: an assistant can look things up, but only you (or your scheduler) decide when to fetch from the API or write files to disk.

FAQ

Will I hit API rate limits?
The Limitless API allows 180 requests per minute. The client honors Retry-After headers and backs off automatically, so even a full --all backfill just slows down instead of failing.
Why are most records category unknown?
By design. The media heuristic is conservative: a log where the wearer spoke is never tagged media, and otherwise it takes two distinct broadcast-style phrases to earn the media tag. Anything ambiguous stays unknown rather than risk mislabeling a real conversation.
My account has records with absurd timestamps. Will they break backfill?
Some accounts contain corrupted lifelogs with epoch-zero (1970) startTimes. The oldest-lifelog anchor used by span and by backfill --all / --from-start ignores anything before 2013 and anchors at the first plausible record instead, so a corrupted record cannot make a backfill walk twenty thousand empty days. To catalog the corrupted records themselves, run ingest orphans: it sweeps them in directly, repairing each start time from the transcript's own line timestamps where possible.
How big does the database get?
Roughly 60-70 KB per recording, including the full transcript, the FTS index, and a verbatim copy of the raw API JSON. A year of heavy pendant use typically lands in the low hundreds of megabytes.
Is my API key or data ever sent anywhere?
No. The key is sent only to api.limitless.ai to fetch your own lifelogs, and everything fetched stays in the local SQLite file. There is no telemetry, no analytics, and no third-party service of any kind.
How do I back up the catalog?
Copy the .db file together with its -wal and -shm siblings while the tool is not running. Or checkpoint first (sqlite3 limitless.db "PRAGMA wal_checkpoint(TRUNCATE)") and copy just the .db. export --format json is also a complete, tool-independent backup.
